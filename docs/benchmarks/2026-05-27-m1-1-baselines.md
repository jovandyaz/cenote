# M1.1 Baselines — Retrieval Quality (2026-05-27)

> **Status: Baseline numbers deferred to `cenote-core 0.2.1` patch release.**

## What ships in 0.2.0

The full M1.1 retrieval stack is functional and unit-tested:

- `VectorRetriever` (voyage-3 + Cohere multilingual embedders)
- `BM25Retriever` with `SpanishTokenizer` (Snowball-ES stemming + accent
  fold + ~440-word stopwords)
- `HybridRetriever` (Reciprocal Rank Fusion, k=60 default, configurable
  `candidate_pool_size`)
- `VoyageReranker` (rerank-2 default) and `CohereReranker`
  (rerank-3.5-multilingual default), both via the shared `_HTTPReranker` base
- `RetrievalBenchmark` orchestrator emitting aggregated +
  per-query `precision@k`, `recall@k`, MRR over an `EvalDataset`
- Bundled datasets:
  - `cenote_mini_es` — 12 queries × 28 docs, Apache 2.0, ships in the wheel
  - `miracl_es_subset` / `miracl_en_subset` — header-only placeholders;
    full subsamples blocked at sprint close (see "Why deferred" below)

## Why deferred

Two independent blockers at sprint close:

1. **MIRACL build failure.** `scripts/build_miracl_subset.py` cannot run
   against current `datasets>=4` because the `miracl/miracl-corpus` HF
   dataset still ships a legacy Python loader script that modern
   `datasets` refuses to execute. Two viable paths for 0.2.1:
   - Pin `datasets<3` in a one-shot env and re-run the build script.
   - Rework the build script to pull MIRACL parquet files directly via
     `huggingface_hub` (more sustainable; survives future `datasets`
     versions).
2. **No API credentials in the release pipeline.** The runner expects
   `VOYAGE_API_KEY` and `COHERE_API_KEY`. These are not available in
   the M1.1 sprint environment; provisioning them is a separate task.

Both blockers are independent of the M1.1 code itself. The retrieval
stack is correct and unit-tested; what's missing is a credentialed,
network-enabled environment with valid MIRACL data.

## Methodology (locked for 0.2.1)

When numbers land, the runner methodology will be:

- Embedder: `voyage-3` (1024-dim).
- Vector store: `InMemoryVectorStore` (cosine similarity).
- Datasets: MIRACL-ES dev subsample (~5000 passages, ~200 queries, CC-BY-SA 3.0),
  MIRACL-EN dev subsample (same shape), `cenote_mini_es` (Apache 2.0, 12
  queries / 28 docs).
- Hardware/runtime: GitHub Actions Ubuntu 24.04, Python 3.12.
- `k=10` for precision/recall/MRR.
- Reranker pipelines fetch top-50 candidates, rerank to top-10.
- BM25 hyperparameters: `k1=1.5`, `b=0.75`.
- RRF `k_constant=60`, default `candidate_pool_size=max(limit*4, 100)`.

**Precision formula note.** `cenote.eval.metrics.precision_at_k` returns
`hits / len(top)` (precision-among-returned), not strict BEIR
`hits / k`. The two converge when the retriever returns ≥k results
per query. Reports use this definition consistently.

## Reproducibility

```bash
VOYAGE_API_KEY=... COHERE_API_KEY=... \
    uv run python -m scripts.run_baseline > /tmp/baseline.json
```

Numbers are deterministic given the same dataset + provider model versions.
Re-running may shift if providers update their default models.

## Caveats

- `cenote_mini_es` is too small (12 queries) for stable headline numbers;
  it exists for sanity-checks of the eval pipeline itself, not as a
  proper benchmark.
- MIRACL subsamples are random with `random.seed(42)`; re-sampling with
  a different seed shifts numbers by ~±3 points.
