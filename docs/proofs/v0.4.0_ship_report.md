## Manual interventions required (one-time, will not recur)

- **release-please PR creation gated by repo setting** — fixed via `gh api` once to flip the `Allow GitHub Actions to create pull requests` setting; no further action needed on subsequent releases.
- **`release.yml` needed `contents: write`** — permissions block patched and committed (`bfe5234`); future releases inherit the fix.
- **`v0.4.0` tag force-updated** — to point at the permissions-fix commit after the original tag landed before the patch. Strictly one-time.
- **`uv.lock` stale after release-please merge** — release-please's PR bumped `pyproject.toml` but not `uv.lock`, so the post-merge `main` had a drift that broke `uv sync --frozen`. Resolved manually via local `uv lock` + push. Pending auto-fix (see Still pending below).

## Automation landed in this same release for v0.5.0+

- **Chained publish job inside `release-please.yml`** using the `release_created` + `tag_name` outputs from `googleapis/release-please-action`. When release-please merges its own PR and creates the tag, the chained job (guard: `if: release_created`) checks out the tag, builds with `uv`, verifies the wheel in an isolated venv, generates the CycloneDX SBOM, attaches it via `softprops/action-gh-release@v3`, and publishes to PyPI through `pypa/gh-action-pypi-publish@release/v1` Trusted Publishing with PEP 740 Sigstore attestations.
- **`release.yml` downgraded to `workflow_dispatch`-only manual fallback** — the `on.push.tags` trigger was removed so the two workflows no longer race on a freshly created tag. Manual reruns on an existing tag still work (all build/verify/SBOM/attach/publish steps unchanged).

## Still pending (TODO for v0.5.0+)

- **Automate `uv lock` on release-please's PR branch.** Two design options are documented in `docs/adrs/0005-release-engineering.md` under the 2026-05-29 v0.4.0 lessons: (a) extra job in the release-please workflow that runs `uv lock` and pushes back to the PR branch; (b) `uv` post-version-bump hook in the release-please config. Option (a) is simpler; option (b) keeps the logic colocated with release-please's bump.
